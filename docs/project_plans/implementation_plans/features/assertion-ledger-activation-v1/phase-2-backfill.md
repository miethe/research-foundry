## THIS PHASE'S SPIKE DECISION IS RESOLVED -- Delivered 2026-07-15

See `docs/project_plans/SPIKEs/assertion-ledger-backfill-mapping.md` (task P2-01) for the full investigation and the parent plan's **"P2-01 SPIKE Verdict"** callout for the summary. Headline finding: a corpus-wide check across all 42 runs with `claims/claim_ledger.yaml` (2,984 extraction facts) found that **97.0% would abstain** at `assertion_materialization.py`'s exact-quote-match gate (`fact_source_quote_mismatch` 93.1% + `missing_exact_passage_quote`-class 3.9%), because the extraction pipeline's `extracted_facts[].text` is a paraphrased summary, while the (pre-fix) materializer requires it to be byte-identical to the source card's verbatim `extracted_points[].quote` -- and this is systemic, not historical debris.

**Resolved decision (P2-01, delivered by the SPIKE document itself, pre-fix estimate)**: adopt **accept-low-yield (option a) + a narrow, spot-checked fuzzy>=0.9 quote-recovery add-on** (re-scoped option b). This raises materializable yield from the 3.0% naive floor to **~6.8% total**, at **+1-2 pts**, not the previously-flagged +3-5 pts. **Rejected**: open-ended gate-relaxation (no generic similarity threshold preserves the fail-closed provenance guarantee -- SPIKE RQ3) and deferring the decision (the evidence is decisive now -- SPIKE RQ4). The SPIKE also corrects a factual assumption in this phase's original task text: option (b) does **not** re-derive against "cached source text" -- no such cache exists anywhere in this workspace (no `assertion_ledger/` directory has ever existed; ledger writes were only just enabled in commit `ba9e551`). The only verbatim substrate for recovery is the per-point `quote` already on each source card. **This ~6.8% figure is superseded below -- it was computed against the pre-fix byte-identity gate before Phase 1.5 landed.**

### Post-Fix Re-Measurement (2026-07-16) -- Two New Blocking Prerequisites Found

Commit `6af82ce` landed Phase 1.5's fix (defects 1a/1b). Re-measuring this phase's backfill yield against the fixed code across all 42 runs (see `docs/project_plans/SPIKEs/assertion-ledger-backfill-mapping.md` -- "## Post-fix re-measurement (2026-07-16)") found the fix itself is a decisive, order-of-magnitude success at the fact level -- **94.78% fact-level yield** (2,835/2,991 supported facts), vs. the pre-fix 3.0%/6.8% estimate above -- but a **third, independent, pre-existing defect (1c)** currently blocks that yield from ever reaching real output: `claim_mapping.py::validate_extraction_fact_claim_mappings()` requires each run's persisted claim ledger to be byte-identical to a fresh re-scan of extraction cards, and rejects any ledger with claims appended after the fact-derived prefix -- which is exactly what the normal report-synthesis flow does when it appends `inference`/`speculation` claims. **39/42 runs (92.9%) abort immediately with `non_bijective_fact_claim_mapping`** before any fact ever reaches the (now-fixed) passage-binding gate. Compounding this, `AssertionMaterializer.materialize_run()`'s all-or-nothing-per-run design means even a single abstaining fact aborts the whole run -- so even with 1c hypothetically fixed, only 8/42 runs (19%) would actually publish anything; the other 34/42 (81%) have at least one abstaining fact each and would still yield zero assertions for the entire run. **Real, end-to-end yield today, calling the unmodified public `materialize_run()` across the whole corpus, is 0.70% (21/2,991)** -- worse than this SPIKE's own pre-fix 3.0% floor, entirely because of defect 1c and the all-or-nothing design, not because the P1.5 fix underperforms.

This adds two new, blocking prerequisite tasks to this phase (**P2-01a**, **P2-01b** below), sequenced ahead of P2-02's write-path driver. Both require careful integrity-gate analysis, not blind relaxation -- see each task's description.

**P2-01's decision is delivered. P2-02 through P2-06 may proceed only after P2-01a and P2-01b land** (new, 2026-07-16) -- both are blocking prerequisites discovered by the post-fix re-measurement, not optional hardening. The residual administrative step (promoting the SPIKE's verdict into a formal design-spec artifact) is folded into P2-01's task below, not a separate blocker.

**State the actual yield explicitly, always -- and it is two numbers, not one.** Prior to the 2026-07-16 re-measurement, this phase stated a single ~6.8% total-yield ceiling. That number is now superseded: **94.78% fact-level yield is achievable once defect 1c (P2-01a) and skip-and-continue materialization (P2-01b) land; 0.70% end-to-end yield is what an unmodified backfill run produces today.** Communicate both numbers to stakeholders -- the fix-isolated ceiling and the current real output -- do not collapse them into one, and do not let the gap between them be discovered at P2-05's dry-run-parity check or later.

## Algorithmic Test Scenario Matrix (H3 requirement)

Per the estimation heuristics (H3: "if you can't enumerate ≥5 scenarios, you don't understand the surface yet"), the following scenarios are drawn directly from `assertion_materialization.py`'s real `_Abstain` codes and must appear in P2-02's/P2-05's/P2-06's test suite:

1. `fact_source_quote_mismatch` -- paraphrased fact text, verbatim quote differs (the dominant real-world case pre-P1.5, 93.1% of the historical corpus; confirm this code's actual post-P1.5 firing rate rather than assuming it is unchanged).
2. `missing_exact_passage_quote` -- source card's evidence point has no `quote` field at all.
3. `unresolved_passage_binding` -- registry has no matching passage for the (source_card_id, quote) pair.
4. `ambiguous_passage_binding` -- registry has more than one matching passage.
5. `missing_source_rights_provenance` / `missing_edition_rights_provenance` -- source card or registry edition lacks `usage`/`allowed_use`.
6. `source_rights_scope_mismatch` -- edition `access_scope` disagrees with the source card's `sensitivity`.
7. `canonical_or_inference_candidate_deferred` -- fact carries a deferred reference field (`canonical_claim_id`, `inference_id`) and must not be materialized as a plain source assertion.
8. Happy path -- fact text exactly equals the source quote (the 3.0% exact-match-eligible case) -- materializes successfully with correct `source_edition_id`/`passage_id` binding.
9. Idempotent re-run -- identical content on 2nd invocation produces `created=False`, same edition/assertion id (registry-layer guarantee; confirm the backfill driver doesn't defeat it).
10. Fuzzy-recovery boundary (P2-06) -- a fact/quote pair at ratio >=0.9 is flagged spot-check-pending, not auto-materialized; a pair at ratio <0.9 (e.g., 0.8) is correctly rejected and abstains, per SPIKE RQ2's finding that 0.8-and-below admits real paraphrase drift.
11. `non_bijective_fact_claim_mapping` adversarial tampering (P2-01a) -- a ledger with a modified/reordered/deleted claim inside the fact-derived prefix must still raise the error, even when legitimate trailing inference/speculation claims are also present; a ledger with only legitimate trailing inference/speculation claims (the normal, expected case, 39/42 real runs) must NOT raise it.
12. Skip-and-continue mixed outcome (P2-01b) -- a run with >=2 facts where at least one materializes and at least one abstains must produce a receipt recording both outcomes, and must not raise/abort the run.
