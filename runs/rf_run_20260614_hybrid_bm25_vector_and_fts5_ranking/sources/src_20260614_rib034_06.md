# Source Card: Normalization processor — OpenSearch Documentation

## Summary
Official OpenSearch documentation for the `normalization-processor`, a search-phase-results processor used to power hybrid (lexical + neural/k-NN) search. It runs between the query and fetch phases, normalizing per-subquery scores onto a common scale (techniques: `min_max`, `l2`, `z_score`) and then combining them (techniques: `arithmetic_mean`, `geometric_mean`, `harmonic_mean`) with optional per-subquery weights. The page also gives tuning guidance grounded in nDCG@10 experiments.

## Key evidence
- (ev_001) Processor runs between query and fetch phases, normalizing and combining scores across query clauses.
- (ev_002) Normalization techniques: min_max (default), l2, z_score.
- (ev_003) Combination techniques: arithmetic_mean (default), geometric_mean, harmonic_mean; z_score supports only arithmetic_mean.
- (ev_004) Per-subquery weights are configurable in [0.0, 1.0], must sum to 1.0; equal weight if omitted.
- (ev_005) BM25 and k-NN scores use different scales, so normalization onto a common scale is beneficial (per experimental data).
- (ev_006) nDCG@10 experiments: size in [100, 200] works best for datasets up to 10M docs; larger sizes only add latency.
