# Source Card: NotebookLM Integration — Sourcing, Reports, Extended Runs, Upload-Back (RF local plan)

## Summary
This internal Research Foundry design plan (draft, 2026-06-13) specifies how Google NotebookLM — a cloud, online-only, non-deterministic engine with no documented REST API — is wired into RF's offline-first, claim-ledger-authoritative pipeline. It maps four use cases (sourcing, running reports, extended runs, upload-back) onto RF's existing seam types and establishes the invariant that NLM output is either a claim-eligible source card or a non-authoritative artifact, never an untagged report claim. It documents the candidate quote-level extraction surface and the reliability tiers of NLM's generation surfaces.

## Key evidence
- (ev_001) NLM has no documented REST API; both CLI and skill drive browser/OAuth automation, so the integration wraps the notebooklm CLI as a subprocess rather than HTTP.
- (ev_002) NLM output is exactly one of two things: a claim-eligible source card or a non-authoritative artifact that never enters the report body untagged.
- (ev_003) All NLM-derived source cards carry usage.requires_network: true and a reliability note because the output is non-reproducible.
- (ev_004) Whether `notebooklm source fulltext --json` yields true quote spans vs summaries is an open question gating evidence fidelity.
- (ev_005) Reliable surfaces are notebooks, sources, chat, mind-map, report, data-table; rate-limited surfaces are audio, video, quiz, flashcards, infographic, slide-deck.
- (ev_006) The integration always writes a deterministic on-disk artifact first and gates the live network call behind available().
- (ev_007) The integration wraps the notebooklm-py CLI v0.3.2 with --json machine output.

## Limitations
- This is a draft internal design plan describing intended/unbuilt behavior, not a record of validated, deployed integration; several claims (e.g., REST API absence, fulltext quote fidelity) are explicitly flagged as open questions or "per findings" rather than confirmed live.
