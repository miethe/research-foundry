# Source Card: File-first vs DB-first architectures for LLM knowledge bases (RIB-037 seed report)

## Summary
This RF-internal seed research report (RIB-037) compares file-first stores (Markdown + Git), DB-first stores (PostgreSQL, Dolt), and a hybrid posture for LLM-readable knowledge bases such as Research Foundry and MeatyWiki. It establishes that Git's content-addressing gives file-first stores the strongest reproducibility/tamper-evidence and that a rebuildable derived index (git-history -> SQLite/Datasette) is the recommended posture, with the single-writer ceiling and an index-amplification cost as the key trade-offs.

## Key evidence
- (ev_001) Recommended posture: Markdown+YAML under Git as single source of truth plus a disposable rebuildable SQLite/Datasette index.
- (ev_002) Git content-addressing makes history self-verifying: any content change changes its key; any commit edit changes every descendant SHA.
- (ev_003) The derived index is a disposable cache with the strongest recovery guarantee; a DB-first store has no external rebuild source.
- (ev_004) File-first/embedded stores share a single-writer ceiling; git-history materializes every version (10 x 30 = 300 rows).
- (ev_005) Obsidian's plain-text vault is editable by any filesystem-aware tool and stock Git; DB-first needs SQL/stored-procedure semantics and a running engine.
- (ev_006) Obsidian is local-first and links notes via bidirectional internal links and a graph view, not a DB schema.
- (ev_007) Speculation: multi-agent swarms likely breach the single-writer breakpoint first, forcing branch+merge isolation or a write coordinator.
