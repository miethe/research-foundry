# Source Card: MCP Authorization Specification (2025-11-25)

## Summary
This is the official, normative Model Context Protocol authorization specification (version 2025-11-25), defining an OAuth 2.1-based authorization flow for HTTP-based MCP transports. It establishes mandatory security boundaries: token audience validation (RFC 8707), a prohibition on token passthrough to upstream APIs, client implementation of Resource Indicators, refresh-token rotation for public clients, and an env-var credential path for STDIO transports. Authorization itself is OPTIONAL for MCP implementations, but when supported HTTP transports SHOULD conform to this spec.

## Key evidence
- (ev_001) Servers MUST validate tokens were issued for them as the intended audience (RFC 8707 Sec 2); invalid/expired tokens MUST get HTTP 401.
- (ev_002) Servers MUST NOT pass the client token through to upstream APIs; the upstream token is a separate token from the upstream authorization server.
- (ev_003) Clients MUST implement RFC 8707 Resource Indicators (resource parameter) in both authorization and token requests, binding the token to the target MCP server.
- (ev_004) Servers MUST only accept tokens intended for themselves and MUST reject tokens not including them in the audience claim.
- (ev_005) Authorization servers SHOULD issue short-lived access tokens and MUST rotate refresh tokens for public clients.
- (ev_006) STDIO-transport implementations SHOULD NOT follow this OAuth spec and instead retrieve credentials from the environment.

## Limitations
- Authorization is OPTIONAL for MCP implementations; the spec is normative only when authorization is supported and (for HTTP transports) when implementers choose to conform.
- The spec implements a selected subset of OAuth 2.1 (IETF draft draft-ietf-oauth-v2-1-13) and references several IETF draft specifications that may evolve.
