# Source Card: SPIFFE Concepts

## Summary
This is the official SPIFFE project page defining the core SPIFFE identity concepts: the SPIFFE ID, the trust domain, the SPIFFE Verifiable Identity Document (SVID), the two SVID encodings (X.509 certificate and JWT token), the SPIFFE Workload API, and the trust bundle. It establishes that workloads prove identity via SVIDs obtained at runtime from the Workload API without pre-provisioned authentication secrets, and that X.509-SVIDs are preferred over JWT-SVIDs to avoid token replay attacks.

## Key evidence
- (ev_001) An SVID is the document with which a workload proves its identity to a resource or caller, carrying a single SPIFFE ID.
- (ev_002) SVIDs come in two formats: an X.509 certificate (X509-SVID) or a JWT token (JWT-SVID).
- (ev_003) X.509-SVIDs are advised over JWT-SVIDs because tokens are susceptible to replay attacks enabling impersonation.
- (ev_004) The Workload API issues identity without requiring the workload to know its identity or hold any authentication token.
- (ev_005) Applications need not co-deploy authentication secrets with the workload, removing static secrets.

## Limitations
- This page is conceptual (the SPIFFE specification surface) and does NOT cover the SOURCE-listed SPIRE details — node attestation, workload attestation by the SPIRE Agent, process-metadata inspection (UID / Kubernetes namespace), automatic SVID rotation timing (~1h), or the SPIRE server/agent runtime architecture. Those claims must be sourced from the SPIRE-specific documentation, not this page; they were omitted here to avoid unsupported evidence.
