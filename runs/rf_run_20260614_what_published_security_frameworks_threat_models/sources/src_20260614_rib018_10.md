# Source Card: Secure multi-tenancy with namespaces | HashiCorp Vault

## Summary
This official HashiCorp Developer tutorial establishes that Vault Enterprise namespaces are logical isolation units that behave as separate, self-serving Vault environments for multi-tenancy. Each namespace owns its own policies, auth methods, secrets engines, tokens, and identity objects, and once signed into a namespace there is no visibility into other namespaces regardless of hierarchy. Namespaces are a commercial feature requiring a Vault Enterprise Standard license or an HCP Vault cluster.

## Key evidence
- (ev_001) Each namespace behaves as an isolated Vault environment.
- (ev_002) Signing into a namespace gives no visibility into other namespaces regardless of hierarchical relationship.
- (ev_003) Tokens, policies, and secrets engines are bound to a namespace; clients need a valid token per namespace, preventing cross-namespace credential reuse.
- (ev_004) Each namespace maintains its own policies, auth methods, secrets engines, tokens, and identity entities/groups.
- (ev_005) Namespaces solve the multi-tenant problem of multiple orgs self-managing secrets on a central Vault.
- (ev_006) Namespaces require a Vault Enterprise Standard license or HCP Vault cluster (a paid, server-backed feature).
- (ev_007) Namespace creation should be done by a highly privileged (root) token.
