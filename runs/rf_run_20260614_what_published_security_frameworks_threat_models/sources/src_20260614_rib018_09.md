# Source Card: Manage dynamic credential leases | HashiCorp Vault

## Summary
This official HashiCorp Vault tutorial documents how Vault's dynamic database credentials are governed by leases. Every credential is bound to a lease with a TTL and a structured lease ID (e.g., `database/creds/readonly/<unique-id>`); credentials remain valid for the lease duration, can be renewed up to a maximum TTL via `vault lease renew`, or are immediately invalidated on revocation. Revocation works per-lease by ID or in bulk via a `-prefix` path match, and once revoked the credentials are no longer valid. This establishes Vault as a server-backed mechanism that brokers and revokes credentials at runtime — a contrast to a file-first governance gate.

## Key evidence
- (ev_001) Each dynamic credential is bound to a lease with a TTL; example shows lease_duration 768h and lease_id format database/creds/readonly/<unique-id>.
- (ev_002) Credentials remain valid for the lease TTL, renewed up to a maximum TTL, or until revoked.
- (ev_003) Leases are renewed before expiry via `vault lease renew <lease_id>`, resetting the TTL.
- (ev_004) A single lease can be revoked immediately by its lease ID without waiting for expiration.
- (ev_005) The `-prefix` flag revokes all valid leases matching a path prefix in bulk.
- (ev_006) Once revoked, the underlying credentials are no longer valid.
- (ev_007) After revocation the lease is no longer valid and is not displayed in listings.

## Limitations
- Tutorial uses illustrative example values (lease IDs, TTLs such as 768h/1h/1m); per-database-role TTL configuration is referenced indirectly rather than stated verbatim on this specific page.
- Source contains an apparent typo "maximum TTO" (intended "maximum TTL"), quoted verbatim in ev_002.
