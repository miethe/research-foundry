# Source Card: Control the concurrency of workflows and jobs - GitHub Docs

## Summary
This official GitHub Actions documentation page specifies the runtime semantics of the `concurrency` keyword at the workflow and job level. It establishes that only one run per concurrency group executes at a time, that additional runs queue as `pending`, and how the `cancel-in-progress` and `queue` properties (single vs max) control cancellation and queue depth. It also documents the recommended per-branch group key, the FIFO ordering caveat, the allowed expression contexts, and the validation error from combining `queue: max` with `cancel-in-progress: true`.

## Key evidence
- (ev_001) Only a single job/workflow per concurrency group runs at a time; the group key is any string or expression.
- (ev_002) A queued run in an active group becomes `pending`; by default an existing pending run is canceled and replaced.
- (ev_003) `cancel-in-progress: true` cancels the currently running job/workflow in the group (also expressible conditionally).
- (ev_004) `queue: single` (default) caps pending at one and replaces it; `queue: max` allows up to 100 pending, canceling extras when full.
- (ev_005) `queue: max` combined with `cancel-in-progress: true` is a workflow validation error.
- (ev_006) Same-group runs are processed FIFO by wait-start time, but ordering is not guaranteed.
- (ev_007) Recommended group key `${{ github.workflow }}-${{ github.ref }}` isolates concurrency per branch; expressions accept github/inputs/vars/needs/strategy/matrix contexts.
