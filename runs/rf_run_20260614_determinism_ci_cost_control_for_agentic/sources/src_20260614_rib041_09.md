# Source Card: How to run evaluations with pytest - LangSmith Docs

## Summary
LangSmith's pytest plugin lets Python developers express datasets and evaluations as ordinary pytest test cases, decorated with `@pytest.mark.langsmith`, with results synced to LangSmith datasets/experiments. For agentic CI cost control it offers on-disk HTTP request caching (`LANGSMITH_TEST_CACHE`) that records requests to a cassettes directory and replays them on later runs, including in CI when the cache is committed. It integrates with standard pytest tooling (pytest-xdist parallelism), auto-logs a `pass` boolean per test, and treats assertion failures as test failures so eval regressions fail the CI job.

## Key evidence
- (ev_001) Setting LANGSMITH_TEST_CACHE caches HTTP requests to disk to avoid re-running expensive LLM calls on every CI commit.
- (ev_002) Cached requests go to tests/cassettes and replay on later runs; committing the cache lets CI reuse it for near-free, deterministic reruns.
- (ev_003) langsmith>=0.4.10 supports host-scoped caching via @pytest.mark.langsmith(cached_hosts=[...]).
- (ev_004) Requires langsmith>=0.3.4, with the langsmith[pytest] extra for rich output and caching.
- (ev_005) Compatible with pytest-xdist for parallel test execution via pytest -n auto.
- (ev_006) A pass/fail boolean is auto-logged under the 'pass' feedback key, and assertions fail the run like normal unit tests.
- (ev_007) Custom feedback (e.g. LLM-judge scores) can be added with log_feedback and traced via trace_feedback.
