# Source Card: pytest-recording (VCR.py-powered HTTP record/replay for pytest)

## Summary
pytest-recording is the official Kiwi.com pytest plugin that wraps VCR.py to record and replay HTTP traffic, exposing a `pytest.mark.vcr` marker tied to VCR.py's `use_cassettes` API. It defaults to VCR's `none` record mode to prevent accidental live network calls, supports record modes (none/once/new_episodes/all and a plugin-specific `rewrite`) via `--record-mode`, and stores recorded request/response pairs as per-test YAML cassettes. It also ships a `pytest.mark.block_network` mark (with `allowed_hosts` whitelisting) to harden tests against unintended network access.

## Key evidence
- (ev_001) pytest-recording is a pytest plugin built on VCR.py that records and replays HTTP traffic for tests.
- (ev_002) The plugin exposes a `pytest.mark.vcr` marker that maps onto VCR.py's `use_cassettes` API for per-test cassette control.
- (ev_003) By default the plugin runs in VCR's `none` record mode specifically to prevent unintentional live network requests during tests.
- (ev_004) Recording is enabled by passing the `--record-mode` CLI option (e.g. `once`) when invoking pytest.
- (ev_005) The plugin adds a `rewrite` record mode that regenerates a cassette from scratch rather than extending it as VCR.py's `all` mode does.
- (ev_006) A `pytest.mark.block_network` mark blocks network access (socket-based transports and pycurl) to give confidence tests do not go over the wire.
- (ev_007) Specific hosts can be whitelisted during network blocking via `allowed_hosts` (regex), e.g. on the mark or via `--allowed-hosts` on the CLI.
- (ev_008) Cassettes are stored as per-test YAML files on disk (capturing the recorded HTTP request/response).
