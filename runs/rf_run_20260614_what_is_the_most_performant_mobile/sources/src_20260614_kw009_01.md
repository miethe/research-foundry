# Source Card: BatchedMesh support for multiDraw*Instanced rendering · Issue #31935 · mrdoob/three.js

## Summary
This open three.js issue (opened 2025-09-24 by andredsm) argues that BatchedMesh should render via the `multiDraw*Instanced` WebGL2 APIs, which it used in earlier versions before that path was deprecated. The current implementation emulates instancing by repeating non-instanced multidraw parameters, which the author benchmarks as slower: roughly 1.5x slower on an integrated Radeon 780M and around 2x slower on a dedicated RTX 4080. Because Firefox lacks `multiDraw*Instanced` support, the request explicitly asks for a Firefox fallback — relevant to mobile and cross-browser deployment decisions.

## Key evidence
- (ev_001) BatchedMesh used multiDraw[Arrays|Elements]Instanced in earlier versions but the feature was deprecated.
- (ev_002) Current BatchedMesh emulates instancing by repeating non-instanced multidraw parameters, which is slower.
- (ev_003) Author-reported speedup of the instanced approach is ~1.5x on iGPU and ~2x on dGPU.
- (ev_004) Benchmarks come from a single machine (Ryzen 9 7940HS, Radeon 780M iGPU, RTX 4080 dGPU).
- (ev_005) Firefox lacks multiDraw*Instanced support; a fallback is requested — a cross-browser/mobile portability constraint.

## Limitations
- Benchmark numbers are author-reported from a single machine (n=1), not independently reproduced; treat the ~1.5x/~2x figures as indicative rather than definitive. Issue is open/unresolved, so it reflects a proposal, not shipped three.js behavior.
