# Source Card: Adreno GPU on Mobile: Best Practices — Game Developer Guide (Qualcomm)

## Summary
Qualcomm's first-party Adreno Game Developer Guide explains how the Adreno GPU's FlexRender hardware switches mid-frame between tile-based (binning/GMEM) and direct (system-memory) rendering, and gives concrete Vulkan guidance to keep work in on-chip tile memory (GMEM). Key levers are correct subpass chaining (executing the full subpass chain per tile to avoid system-memory resolves, "gains of over 10% frametime"), the VK_QCOM_tile_memory_heap extension to keep deferred-rendering intermediates resident in GMEM, minimizing/merging render passes, and invalidating framebuffer contents early to avoid wasteful GMEM resolves. The throughline is that any pass structure that breaks the tile/subpass chain forces GMEM-to-system-memory traffic, a measurable bandwidth and battery cost on Adreno.

## Key evidence
- (ev_001) Adreno FlexRender switches mid-frame between binning/GMEM and direct/system-memory mode; optimize for both.
- (ev_002) FlexRender direct-mode triggers include heavy vertex-shader texture sampling, few vertices/draws, and tessellation/geometry shaders.
- (ev_003) Per-tile execution of the full subpass chain avoids system-memory resolves after each pass; merged subpasses yield gains of over 10% frametime.
- (ev_004) VK_QCOM_tile_memory_heap keeps deferred-rendering images/buffers resident in GMEM across multiple passes, saving bandwidth.
- (ev_005) Invalidating framebuffer contents early avoids wasteful GMEM-to-system-memory resolves.
- (ev_006) Minimize render passes by combining consecutive passes that share the same color buffer.
