# Source Card: Asynchronous Inference - Ray Serve

## Summary
Official Ray Serve documentation (Ray 2.56.0) describing an asynchronous-inference pattern for long-running workloads. Incoming HTTP requests enqueue work into a background queue and return immediately; a `@task_consumer` Serve deployment pulls and executes tasks from that queue via `@task_handler` methods, decoupling request lifetime from compute time. Clients poll task status (`get_task_status_sync`) rather than blocking, concurrency/backpressure is bounded per replica by `max_ongoing_requests`, and a built-in autoscaling policy scales workers to queue depth.

## Key evidence
- (ev_001) Long-running inference runs as a background task so HTTP APIs stay responsive.
- (ev_002) Work exceeding HTTP timeouts is enqueued and the request returns immediately.
- (ev_003) Async execution decouples request lifetime from compute time.
- (ev_004) `@task_consumer` is a Serve deployment that consumes and executes queued tasks.
- (ev_005) `max_ongoing_requests` provides per-replica backpressure; failures retry then hit a DLQ.
- (ev_006) Clients poll status via `get_task_status_sync` instead of blocking synchronously.
- (ev_007) Built-in autoscaling policy scales replicas to queue length + in-flight demand.

## Limitations
- Ray Serve currently supports only synchronous handlers (an `async def` handler raises `NotImplementedError`); the "async" refers to decoupled request handling, not async task bodies. Terminology note: the framework uses `max_ongoing_requests` for concurrency capping (not a `max_queued_requests` knob). Docs track the "latest" (Ray 2.56.0) build; API names may shift across versions.
