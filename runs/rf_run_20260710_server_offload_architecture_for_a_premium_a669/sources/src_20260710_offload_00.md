# Source Card: TN3193: Managing the on-device foundation model's context window

## Summary
This Apple Developer technote documents the fixed 4096-token context window of Apple's on-device foundation model (per LanguageModelSession) and how to budget for it. It explains that every input and output element — instructions, prompts, tool schemas, Generable JSON schemas, @Guide descriptions, and model responses — consumes the shared window, and prescribes concrete workarounds: measuring with tokenCount(for:), splitting oversized tasks across sessions with carried-forward summaries, using RAG for large corpora, and catching the .exceededContextWindowSize error.

## Key evidence
- (ev_001) Apple's on-device foundation model has a 4096-token context window per LanguageModelSession.
- (ev_002) A token is ~3-4 chars in English but ~1 char per token for Chinese/Japanese/Korean.
- (ev_003) All input and output count against the window: instructions, prompts, tool schemas/IO, Generable schemas, and responses.
- (ev_004) tokenCount(for:) measures token consumption before sending.
- (ev_005) Split large tasks into chunked sessions, carrying the prior summary forward.
- (ev_006) For large knowledge bases, use RAG with Core ML / Natural Language embeddings.
- (ev_007) Vectorizing a large corpus is computationally heavy — run it as separate data-prep or on a server.
- (ev_008) Exceeding the window throws .exceededContextWindowSize, which must be handled.

## Limitations
- Scope is Apple's on-device Foundation Models framework specifically (iOS/macOS Apple Intelligence, 2025); token ratios and the 4096-token limit are model/hardware-specific and may change across OS versions. Not generalizable to cloud LLMs.
