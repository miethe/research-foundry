# Source Card: Tracking issue: ANN (Approximate Nearest Neighbors) Index · Issue #25 · asg017/sqlite-vec

## Summary
This GitHub tracking issue, authored by sqlite-vec maintainer asg017, documents the plan to move beyond v0.1.0's brute-force-only vector search, which slows on large datasets (>1M vectors with large dimensions). The stated goal is to add an approximate-nearest-neighbors (ANN) mode before v1, trading accuracy and resource usage for speed, while staying SQLite-native (shadow tables, page-sized data, low memory). DiskANN — especially LM-DiskANN — is the favored first algorithm, with remaining design questions coordinated in this single issue.

## Key evidence
- (ev_001) sqlite-vec v0.1.0 is brute-force search only and slows on >1M vectors with large dimensions.
- (ev_002) Goal is to add ANN before v1, trading accuracy/resource usage for speed.
- (ev_003) ANN must be SQLite-native: shadow tables, data fitting in pages, low memory usage.
- (ev_004) DiskANN (notably LM-DiskANN) is favored for its simplicity over IVF/HNSW.
- (ev_005) Open ANN design questions are coordinated in this single GitHub tracking issue.

## Limitations
- This is a design/tracking issue, not finalized implementation: no final algorithm or schema decision is recorded. State reflects the v0.1.0 era (issue posted 2024-06-21); the actual ANN implementation may have evolved since.
