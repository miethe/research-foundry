# Source Card: IL2CPP build size optimizations

## Summary
This first-party Unity Support article explains why the IL2CPP scripting backend can produce larger app binaries than Mono (AOT C#-to-C++ compilation, and universal builds bundling both 32-bit and 64-bit slices) and gives guidance for measuring and reducing iOS/tvOS binary size against Apple's submission limits. Its concrete recommendations are around stripping behavior (IL2CPP always strips byte code; set Stripping Level to Disabled) and using the MapFileParser utility to attribute size. The article is iOS/tvOS-scoped and dated to the Unity 5.x / XCode 7 / iOS 9 era, so it does not address Android APK baselines, "Strip Engine Code," or glTF runtime ingestion.

## Key evidence
- (ev_001) IL2CPP always strips byte code regardless of Stripping Level; Unity recommends setting Stripping Level to Disabled.
- (ev_002) IL2CPP AOT-compiles C# to C++ into the binary, which can be larger than the Mono backend.
- (ev_003) Universal (fat) builds bundle 32-bit and 64-bit slices, roughly doubling binary size.
- (ev_004) Apple's binary size limits vary by MinimumOSVersion (80 MB / 60 MB per slice / 500 MB).
- (ev_005) Unity's MapFileParser utility attributes executable size to individual scripts and engine code.
- (ev_006) Article is scoped to Unity 5.2.0p1+, XCode 7+, iOS 9.0+.

## Limitations
- Source is iOS/tvOS-focused: it covers Apple binary slices and submission limits, not Android APK baseline sizes, and never mentions a "Strip Engine Code" option, glTF/glTFast runtime ingestion, or Filament comparisons. The prompt's supplied key points about Android baselines (~6-17MB), react-native-filament (~4MB), and glTFast are NOT supported by this article's text and were omitted.
- Undated ("2 years ago Updated"); references Unity 4.6.x-5.4.x and XCode 6-7, so figures and behavior are dated relative to current Unity releases.
