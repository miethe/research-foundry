# The KnitWit Crochet IR: An Evidence-Grounded Schema, Validation-Rule Set, and Adopt/Adapt Decision over CrochetPARADE and CrochetBench

## Executive summary

This memo specifies an evidence-grounded intermediate representation (IR) for KnitWit's crochet engine by reconciling four peer-reviewed crochet-modeling sources (Digital Crochet/Seitz, AmiGo, Guo stitch meshes, Greer/Mould force-directed layout) with two production artifacts (the CrochetPARADE DSL and the CrochetBench benchmark). The recommended design is a hybrid: a node-and-edge graph core for validation under a CrochetPARADE-style DSL surface for human authoring and corpus compatibility. **Inference:** The KnitWit Crochet IR should be a DSL surface over a graph core (hybrid), because three independent peer-reviewed sources (Digital Crochet, AmiGo, and Greer/Mould) all model crochet natively as a node-and-edge graph while CrochetPARADE's adoptable public-domain layer is itself a node/connection grammar, so a graph core gives a single validation substrate and a CrochetPARADE-style DSL gives the human-authoring and corpus-compatibility surface. [claim:clm_inf01]

The expressiveness of a CrochetPARADE-grammar IR is empirically anchored: CrochetBench compiled 6,085 real crochet patterns into the CrochetPARADE DSL and validates structural correctness by compiling outputs in the CrochetPARADE interpreter, demonstrating the DSL is expressive enough to capture thousands of real patterns. [claim:clm_066] The same grammar is the natural textual target for AmiGo's `sc`/`inc(x)`/`dec(x)` instruction output. **Inference:** The IR is well-positioned to round-trip real patterns losslessly, because the same CrochetPARADE DSL that CrochetBench compiled 6,085 real patterns through is also the natural textual target for AmiGo's sc/inc(x)/dec(x) instruction output, so a CrochetPARADE-grammar IR can simultaneously be the forward engine's input and the inverse generator's output without a representation change. [claim:clm_inf11]

An explicit, deterministic validator is not optional for a machine-generation pipeline. **Inference:** An explicit executable IR with a deterministic validator is essential rather than optional for KnitWit, because CrochetBench shows even the strongest VLMs produce only 5-8% executable project-level programs and DINO similarity of 0.10-0.17, so machine-generated patterns cannot be trusted without the IR's hard validity gates catching the roughly 92-95% that would otherwise be malformed. [claim:clm_inf17]

The licensing finding cleanly separates three layers: the public-domain grammar, the GPLv3 engine, and the Yarnspirations-copyright corpus. **Inference:** The cleanest legally-safe adoption path is a three-way separation: implement a clean-room parser/validator for the public-domain CrochetPARADE grammar (avoiding the GPLv3 engine code), do not vendor or link the GPLv3 engine into KnitWit's app, and use the Yarnspirations-copyright CrochetBench corpus only as a private validation/test oracle (never redistributed in the product), keeping KnitWit's app license unconstrained by copyleft. [claim:clm_inf12]

## Source landscape

CrochetPARADE is an acronym for 'Crochet PAttern Renderer, Analyzer, and DEbugger,' authored by Svetlin Tassev, and is a platform for creating, visualizing, and analyzing both 2D and 3D crochet patterns. [claim:clm_072] CrochetPARADE uses a custom DSL grammar to remove the ambiguities of plain-English crochet instructions, parsing and checking patterns for correctness before building a 3D virtual model. [claim:clm_011] The custom grammar is intended to ensure accuracy and precision in instructions, avoiding the ambiguities of plain-English crochet directions. [claim:clm_074] The CrochetPARADE grammar follows rules similar to those of real programming languages, making it usable to teach both crocheting and programming. [claim:clm_012] CrochetPARADE is built on the SVG.js and three.js JavaScript libraries (three.js handles the 3D rendering). [claim:clm_079] The repository is hosted on Codeberg (canonical) and mirrored on GitHub, with showcase asset filenames timestamped 2025-02-22, indicating active maintenance in early 2025. [claim:clm_015]

The paper proposes the first visual, domain-specific, graph-based language for unambiguous crochet pattern representation, contributed alongside a 2D-editing/3D-viewing editor prototype and a qualitative user evaluation. [claim:clm_016] Digital Crochet (Onward! 2022) defines a graph-based crochet IR where increases are nodes with multiple incoming edges and decreases nodes with multiple outgoing edges, and uses a force-directed algorithm to lay out stitches in 3D as the pattern is built. [claim:clm_069] AmiGo (SCF 2022) is the state-of-art inverse mesh-to-pattern pipeline: it segments a closed mesh into crochetable components, places increases on positive curvature and decreases on negative curvature, and uses join-as-you-go to attach pieces without sewing; no public code (Matlab/C++). [claim:clm_068] Guo et al. 2020 (SCF) define a special 'current loop' edge type that tracks the live yarn loop moving from stitch to stitch, providing the basis for guaranteeing one continuous yarn path in generated patterns. [claim:clm_070] CrochetBench adopts the CrochetPARADE DSL as its intermediate representation for crochet patterns. [claim:clm_043]

| Stitch-extension DEF | The language is extensible to new stitch types via a DEF directive carrying a raw node/connection grammar of the form top_nodes:bottom_nodes~attachments:other_nodes:connections. | Adopt as-is (extension carrier) | [claim:clm_032] |

**Inference:** CrochetPARADE's repeat grammar (bracketed blocks with integer multiplier, plus the early-end and late-start angle markers and line-wrap) should be adopted as-is for the IR's repeat/round surface syntax, since it already losslessly encodes the partial-repeat edge cases that real Yarnspirations patterns require, evidenced by CrochetBench compiling 6,085 patterns through it. [claim:clm_inf04] **Inference:** The layers abstraction (rows and rounds unified as a per-node integer property, continuous spiral vs joined rounds distinguished only by where the next stitch is inserted) should be adopted verbatim as the IR's round/row model, because two independent graph papers (Digital Crochet and Greer/Mould) and CrochetPARADE's keyword-free line-as-row convention all reach the same no-structural-distinction design. [claim:clm_inf07]

[claim:clm_inf16] **Inference:** CrochetPARADE's stitch identification by type+row+position is insufficient for the visualizer's playback/highlighting needs and must be augmented with stable per-stitch GUIDs, because position-based IDs change whenever a row is edited whereas the graph core already gives every stitch a persistent node identity that GUIDs can ride on. [claim:clm_inf15]

## Amigurumi extensions the literature does NOT cover

**Inference:** Amigurumi piece boundaries and assembly joints are the clearest gap the prior art does not cleanly cover: CrochetPARADE only offers a low-level start_anew/skip disjoint marker, AmiGo's join-as-you-go has the right semantics but no public code or text DSL, and Greer/Mould explicitly call piece-joining ad hoc with no systematic method, so the IR must add first-class piece-boundary and joint (join-as-you-go vs sew) constructs as an original extension. [claim:clm_inf05] Stated limitations include the assumption of constant edge length 1 (real stitch dimensions vary with tension, hook, and yarn, so a swatch measurement is needed for accuracy), testing only on beginner-level patterns and sc-like stitches, and ad hoc piece-joining with no systematic method yet. [claim:clm_027] **Inference:** The IR should adopt AmiGo's join-as-you-go coupling rule as the formal contract for the new assembly-joint construct: a join joint is legal only when the attaching segment's first row is coupled to the host segment's last row (matching column edges), which gives the otherwise-undefined extension a machine-checkable validation rule grounded in AmiGo's existence theorem. [claim:clm_inf06]

**Inference:** Amigurumi-critical operations (magic ring, invisible decrease, continuous-spiral vs joined-round modes) are NOT named primitives in any reviewed source and must be added as the IR's v1 amigurumi extension, expressible through CrochetPARADE's DEF mechanism for the stitch-level ones (invisible decrease as a dec variant) and Greer's shared anchor/ring-node convention for the magic ring. [claim:clm_inf14] Rows and rounds are treated identically ('rows' is used generically for both), and the first row's single-crochet nodes all connect to a shared anchor (ring) node while later rows work into the previous row. [claim:clm_025]

**Inference:** Gauge/yarn-diameter/scale should be parameterized in the IR by a single primary stitch-width parameter w with an optional per-stitch-type aspect-ratio override, because AmiGo controls gauge with exactly one width w mapping mesh area to stitch count, Guo's pipeline uses a single stitch aspect ratio, and both note that one global parameter limits accuracy across stitch types, justifying the override hook. [claim:clm_inf10] Gauge is controlled by a single stitch width w: the row function f and column function g are uniformly sampled on a 2D grid of width w, producing vertices of S indexed (f/w, g/w), so one width parameter directly maps mesh area to stitch count. [claim:clm_062] The pattern-generation system (a variant of Narayanan et al. 2019) uses a single stitch 'aspect ratio' to drive the remeshing process, a gauge-like parameter that limits accuracy across varied stitch types. [claim:clm_042]

| Single-continuous-yarn (Seitz) | Because a crochet pattern is worked along a single yarn, the graph encodes a crochetability (reproducibility) constraint: edge orderings must be such that no loop ever uses an insertion point that has not yet been created, guaranteeing exactly one unambiguous yarn path through the pattern. | Seitz crochetability | [claim:clm_053] |
| No-use-before-create | The core validity invariant for crochetability is that no node may be worked into before it is created (no use of a future insertion point), and the editor renders the graph in 2D/3D via a force-based / 3D force-directed layout heuristic. | Seitz crochetability rules | [claim:clm_021] |

**Inference:** Per-round stitch-count invariants and inc/dec legality are directly checkable in the IR by counting node out/in-degree per layer and validating against the declared count, because CrochetBench reports CrochetPARADE already flags mismatched stitch counts and impossible attachments and AmiGo emits inc/dec from vertex connectivity counts, so the count invariant is a re-use of existing validator behavior rather than new machinery. [claim:clm_inf08] **Inference:** The single-continuous-yarn / current-loop continuity invariant is the most strongly corroborated rule in the corpus, asserted in three forms (Seitz's no-use-before-create, AmiGo's row-coupling existence theorem, and Guo's current-loop edge), and therefore must be a hard, non-bypassable gate in the validator for both forward and inverse engines. [claim:clm_inf03] **Inference:** Graph branching for shaping should be canonicalized as inc = one base insertion point with multiple feeding stitch nodes and dec = one stitch node consuming multiple base insertion points, because Digital Crochet, AmiGo's transducer, and Guo's stitch-mesh faces independently encode exactly this multiplicity, making it the cross-source-validated representation for the IR's increase/decrease semantics. [claim:clm_inf02]

**Inference:** CrochetPARADE's built-in too-loose/too-tight stitch detection maps to a geometric/gauge-plausibility check that is distinct from and should run after the topological validity gates, because it is a force-directed convergence heuristic (author-reported only to within ~10% of requested lengths) and thus cannot be treated as an exact correctness oracle the way the continuity and count invariants can. [claim:clm_inf09] The force-directed engine does not produce exact stitch lengths; the author reports it converges to within 10% of requested stitch lengths under default settings. [claim:clm_035] **Speculation:** A bounded inc/dec-rate rule (e.g. no more than roughly a 2x stitch-count change per round, with tighter limits near high curvature) will likely be needed for amigurumi crochetability, but the reviewed corpus supplies only AmiGo's qualitative curvature-obstruction table and sampling-rate modulation rather than a numeric per-round cap, so the exact threshold must be derived empirically and is currently speculation. [claim:clm_spec01]

## Graph-vs-DSL derivation

The two native crochet representations in the corpus are the graph IR and the textual DSL, and they are not competing but layered. The graph IR represents each stitch as a node, with horizontal edges encoding sequential connections (one stitch made immediately after another) and vertical edges encoding working connections (top stitch worked into the bottom stitch). [claim:clm_023] In the language a stitch is represented as a graph node (an 'insertion point'), and the basic idea is to directly encode the structure and dependency relations of a pattern rather than a linear sequence of stitches. [claim:clm_017] Edges encode the worked-into/insertion relationship between stitches: 'previous' edges trace the yarn origin and each node has exactly one outgoing previous edge, while 'insertion' edges point to the insertion points that the current stitch is worked into. [claim:clm_018] Increases are represented as multiple incoming insertion edges into one insertion point, and decreases as multiple outgoing insertion edges from a single stitch node. [claim:clm_019]

Seitz's layer model is the round/row unifier the IR should inherit. Rows and rounds are unified into a single concept the authors call 'layers,' requiring no structural change in the graph because whether a pattern is worked row-wise or round-wise depends only on where the next stitch is inserted; each node stores its layer number as a property. [claim:clm_051] A projectional editor prototype lets designers build patterns step-wise 'as if crocheting,' which by construction only permits reproducible (crochetable) patterns, and renders the same graph as a 2D crochet chart and a force-based 3D view. [claim:clm_054] The language and editor were evaluated in a think-aloud study with six professional crochet designers, recruited from 25 respondents of a formative survey originally sent to 200 myboshi designers, who confirmed the language could express 2D and 3D patterns and remove ambiguities present in standard crochet charts. [claim:clm_055] In a think-aloud study with six professional crochet designers, all but one participant successfully reproduced the pattern in 2D; four cited the shape preview as beneficial, and several expressed enthusiasm about the 3D perspective for understanding shape (e.g., amigurumi proportions). [claim:clm_022]

AmiGo formalizes the same graph at the geometry level. The Crochet Graph G = (S, R union C) models a pattern as vertices S that are stitch tops/bases indexed (i,j) for the j-th stitch in row i, column edges C representing stitch stems between rows, and row edges R representing within-row connectivity. [claim:clm_057] Curved geometry is achieved in crochet through stitches that locally increase (inc(x)) or decrease (dec(x)) the stitch count by x, so curvature is encoded directly as stitch-level operations. [claim:clm_058] Multi-part shapes use 'join-as-you-go': each segment is crocheted directly onto the last row of the previous segment so no sewing is needed, segment order follows a topological sort of the segment DAG, and any shared edge (s,t) requires the last row of M_s to be coupled to the first row of M_t. [claim:clm_061] For regions of negative Gaussian curvature the column sampling rate is modulated by setting <grad g, J grad f> = h(k_j grad f) with h(x)=tanh(-x/alpha)/2+1 and alpha=10, to avoid degenerate sampling rates. [claim:clm_063]

The Greer/Mould force-directed work confirms the same node/edge primitives and the row=round convention. Rows and rounds are treated identically ('rows' is used generically for both), and the first row's single-crochet nodes all connect to a shared anchor (ring) node while later rows work into the previous row. [claim:clm_025] Shaping effects done without crochet stitches (e.g. pulling yarn tight to create a dimple) are encoded as additional 'constraint edges', as in the apple model where a constraint edge of length 6 (chosen by trial and error) links the first and last node. [claim:clm_024] The layout is a non-linear least-squares optimization (solved with Ceres Solver) blending an edge-length energy (targeting unit-length edges) and a curvature energy as (1-lambda)*EL + lambda*C, with a default lambda=0.65 (raised to lambda=0.9 for the bunny head and body); inflating the planar graph emulates stuffing. [claim:clm_026]

The stitch-mesh paradigm supplies the continuity-edge vocabulary. The paper represents crochet patterns in the stitch-mesh paradigm as a library of tiles where each tile contains yarn geometry and tiles connect along their edges. [claim:clm_036] Faces use short-term (orange) edge types crossed by the leading loop and free yarn and long-term (purple) edge types crossed only by loops, encoding crochet's construction dependencies. [claim:clm_038] Four edge labels - Next/Previous (consecutive-stitch leading-loop dependencies) and Future/Past (loop-through-fabric dependencies) - connect as previous-next and past-future pairs to cover a surface. [claim:clm_039] Decreases use multi-past-edge faces (sc2tog uses two past loop edges to make one future loop edge; sc3tog three), while the 'inc' utility face splits a future edge into two future edges, and 'turn'/'cap'/'ch_edge' utility faces enable partial-width rows and variants. [claim:clm_040] An automatic pipeline (a variant of Narayanan et al. 2019) turns a manifold triangulated 3D mesh into a stitch mesh of quad faces (sc/turn), pentagons (sc2tog), and increase triangles (inc). [claim:clm_041]

## Round-trip expressiveness

The expressiveness claim rests on a measured fact and an output-format match. CrochetBench is a dataset of 6,085 crochet patterns across 55 project categories scraped from Yarnspirations, with roughly 98% image coverage. [claim:clm_080] CrochetBench compiled 6,085 real crochet patterns into the CrochetPARADE DSL and validates structural correctness by compiling outputs in the CrochetPARADE interpreter, demonstrating the DSL is expressive enough to capture thousands of real patterns. [claim:clm_066] The benchmark vendors the CrochetPARADE repository by cloning it from Codeberg for validation. [claim:clm_044] Functional evaluation compares the target product image against a rendering produced from each model's executable program using DINO similarity. [claim:clm_083]

On the inverse side, AmiGo's transducer emits exactly the `sc`/`inc(x)`/`dec(x)` vocabulary that the CrochetPARADE DSL accepts. Increases and decreases are emitted by a transducer that reads vertex connectivity: x>1 vertices in row S_i feeding one head vertex in S_{i+1} produce dec(x), and x>1 vertices in S_{i+1} feeding one vertex in S_i produce inc(x). [claim:clm_059] **Inference:** The IR is well-positioned to round-trip real patterns losslessly, because the same CrochetPARADE DSL that CrochetBench compiled 6,085 real patterns through is also the natural textual target for AmiGo's sc/inc(x)/dec(x) instruction output, so a CrochetPARADE-grammar IR can simultaneously be the forward engine's input and the inverse generator's output without a representation change. [claim:clm_inf11] **Speculation:** If KnitWit ships the forward visualizer on the hybrid IR first and treats the AmiGo-style inverse mesh-to-pattern generator as a later experiment-gated phase, the IR's graph core plus coupling/continuity validator will likely be reusable unchanged across both engines, making the inverse generator a payoff add-on rather than an IR redesign. [claim:clm_spec02]

The benchmark also quantifies why a deterministic IR is load-bearing. The benchmark defines four tasks: A Stitch Recognition (6,009), B Instruction Selection (6,003), C Instruction Generation (6,009), and D Instruction-to-DSL (119 step-level / 100 project-level). [claim:clm_049] Even the strongest tested models produce only 5-8% executable programs at the project level, showing current VLMs largely fail at executable generation. [claim:clm_084] DINO similarity scores of valid model renderings remain uniformly low (0.10-0.17), far below the threshold indicating faithful reproduction. [claim:clm_085] Step-level validity is also low, with most models achieving under 15% validity even in the first two steps of generation. [claim:clm_086] Across all tasks, performance sharply decreases as evaluation shifts from surface-level similarity to executable correctness, revealing limits in long-range symbolic reasoning and 3D-aware procedural synthesis. [claim:clm_087] **Speculation:** Designer-defined / non-standard stitches will likely be the long-tail blocker for the corpus round-trip claim, because Guo's tile set covers only ch/ss/sc/sc2tog/sc3tog and Greer tested only sc-like stitches, so CrochetPARADE's DEF extensibility mechanism will probably have to carry most real-world stitch variety (dc, tr, post stitches) that the graph-layout literature has not yet validated. [claim:clm_spec03]

## Licensing finding

The three layers carry distinct licenses that must not be conflated. The CrochetPARADE manual is CC BY-NC-SA and the computational components are GPLv3, but the grammar itself cannot be copyrighted and is in the public domain. [claim:clm_007] The CrochetPARADE website and all of its computational components are released as free and open source under the GPLv3 license. [claim:clm_008] The user manual, which includes a description of the CrochetPARADE grammar, is released under the Creative Commons BY-NC-SA license. [claim:clm_009] The grammar itself cannot be copyrighted and is in the public domain, so the DSL grammar is adoptable even though the reference engine is GPLv3 copyleft. [claim:clm_010] The CrochetPARADE grammar/language itself is explicitly stated to be in the public domain and not copyrightable, separating the language from the GPLv3-licensed implementation. [claim:clm_033] CrochetPARADE is an open-source forward crochet visualizer whose engine is GPLv3 (copyleft) but whose pattern-language grammar is public domain, making the grammar adoptable as a Crochet IR while the engine carries integration risk. [claim:clm_065]

The corpus carries a separate copyright constraint. The CrochetBench tooling is permissively licensed (MIT, assumed) but the underlying patterns are copyrighted by Yarnspirations, so the corpus is usable for validation/testing rather than redistribution. [claim:clm_067] To respect copyright the authors do not redistribute raw pattern PDFs or full text, releasing only GPT-generated structured JSON annotations, reference URLs, and their scripts. [claim:clm_081] The implementation behind the force-directed layout work is released as open-source code at https://github.com/EnbyMonkey/modelingamigurumi.git, and the paper is an open-access article under the Creative Commons Attribution License. [claim:clm_028]

**Inference:** The cleanest legally-safe adoption path is a three-way separation: implement a clean-room parser/validator for the public-domain CrochetPARADE grammar (avoiding the GPLv3 engine code), do not vendor or link the GPLv3 engine into KnitWit's app, and use the Yarnspirations-copyright CrochetBench corpus only as a private validation/test oracle (never redistributed in the product), keeping KnitWit's app license unconstrained by copyleft. [claim:clm_inf12] **Inference:** Adopting the CrochetPARADE grammar carries low copyleft risk but a real reimplementation cost, since the public-domain grammar can be freely cloned yet the only reference implementation that proves it works (the engine that CrochetBench vendors from Codeberg) is GPLv3, so KnitWit must rebuild the parser/validator from the manual rather than reuse the tested engine. [claim:clm_inf13]

## Recommendations and decision rules

The IR is a DSL surface over a graph core. **Inference:** The KnitWit Crochet IR should be a DSL surface over a graph core (hybrid), because three independent peer-reviewed sources (Digital Crochet, AmiGo, and Greer/Mould) all model crochet natively as a node-and-edge graph while CrochetPARADE's adoptable public-domain layer is itself a node/connection grammar, so a graph core gives a single validation substrate and a CrochetPARADE-style DSL gives the human-authoring and corpus-compatibility surface. [claim:clm_inf01] The report's sequencing guidance is to ship the forward visualizer first on a CrochetPARADE-style IR (lower risk, immediate value) and pursue the inverse (3D->pattern) generator, which is more complex, as a longer-term payoff de-risked by focused experiments. [claim:clm_071]

The continuity gate is the single non-bypassable rule. **Inference:** The single-continuous-yarn / current-loop continuity invariant is the most strongly corroborated rule in the corpus, asserted in three forms (Seitz's no-use-before-create, AmiGo's row-coupling existence theorem, and Guo's current-loop edge), and therefore must be a hard, non-bypassable gate in the validator for both forward and inverse engines. [claim:clm_inf03] The deterministic validator is mandatory, not optional. **Inference:** An explicit executable IR with a deterministic validator is essential rather than optional for KnitWit, because CrochetBench shows even the strongest VLMs produce only 5-8% executable project-level programs and DINO similarity of 0.10-0.17, so machine-generated patterns cannot be trusted without the IR's hard validity gates catching the roughly 92-95% that would otherwise be malformed. [claim:clm_inf17]

Branching, piece boundaries, gauge, and stitch identity each have a fixed decision rule. **Inference:** Graph branching for shaping should be canonicalized as inc = one base insertion point with multiple feeding stitch nodes and dec = one stitch node consuming multiple base insertion points, because Digital Crochet, AmiGo's transducer, and Guo's stitch-mesh faces independently encode exactly this multiplicity, making it the cross-source-validated representation for the IR's increase/decrease semantics. [claim:clm_inf02] **Inference:** The IR should adopt AmiGo's join-as-you-go coupling rule as the formal contract for the new assembly-joint construct: a join joint is legal only when the attaching segment's first row is coupled to the host segment's last row (matching column edges), which gives the otherwise-undefined extension a machine-checkable validation rule grounded in AmiGo's existence theorem. [claim:clm_inf06] **Inference:** Gauge/yarn-diameter/scale should be parameterized in the IR by a single primary stitch-width parameter w with an optional per-stitch-type aspect-ratio override, because AmiGo controls gauge with exactly one width w mapping mesh area to stitch count, Guo's pipeline uses a single stitch aspect ratio, and both note that one global parameter limits accuracy across stitch types, justifying the override hook. [claim:clm_inf10] **Inference:** CrochetPARADE's stitch identification by type+row+position is insufficient for the visualizer's playback/highlighting needs and must be augmented with stable per-stitch GUIDs, because position-based IDs change whenever a row is edited whereas the graph core already gives every stitch a persistent node identity that GUIDs can ride on. [claim:clm_inf15]

The gauge-plausibility check runs after the topological gates. **Inference:** CrochetPARADE's built-in too-loose/too-tight stitch detection maps to a geometric/gauge-plausibility check that is distinct from and should run after the topological validity gates, because it is a force-directed convergence heuristic (author-reported only to within ~10% of requested lengths) and thus cannot be treated as an exact correctness oracle the way the continuity and count invariants can. [claim:clm_inf09]

## Open questions

- What numeric per-round inc/dec-rate cap (and curvature-dependent tightening) actually keeps amigurumi crochetable, given the corpus supplies only a qualitative curvature-obstruction table [claim:clm_spec01]?
- Will the CrochetPARADE DEF mechanism carry the full real-world stitch variety (dc, tr, post stitches) that the graph-layout literature has not yet validated [claim:clm_spec03]?
- Can the hybrid IR's graph core plus coupling/continuity validator truly be reused unchanged across the forward visualizer and the AmiGo-style inverse generator [claim:clm_spec02]?
