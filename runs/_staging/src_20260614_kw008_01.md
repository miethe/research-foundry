# Source Card: Filament Issue #1445 — Question: correct way to update vertex buffer data

## Summary
This official google/filament GitHub issue (2019) asks whether mutating vertex data each frame via `VertexBuffer::setBufferAt()` is the correct approach for a dynamically changing mesh, or whether the whole VertexBuffer must be re-created. A Filament maintainer (prideout) confirms that calling `setBufferAt` to mutate an existing vertex buffer in place is perfectly fine, and clarifies that this is different from the (separately debated) ability to swap a renderable from one vertex buffer to another. The thread establishes the supported per-frame vertex-update path for deforming meshes and point clouds.

## Key evidence
- (ev_001) Per-frame vertex updates use `VertexBuffer::setBufferAt(engine, slot, BufferDescriptor(data, size))` over a fixed-location buffer whose contents change.
- (ev_002) This pattern supports a dynamically changing mesh without rebuilding the VertexBuffer each frame.
- (ev_003) A Filament maintainer confirms in-place `setBufferAt` mutation is a correct, supported usage.
- (ev_004) In-place buffer mutation is distinct from swapping a renderable between two different vertex buffers (debated in #1279).

## Limitations
- Single short Q&A from 2019; later Filament releases may have added/changed APIs (e.g., RenderableManager::setGeometryAt) — those specific APIs are NOT documented by this issue and must be sourced elsewhere before being asserted.
- The issue confirms vertex-buffer mutation only; per-renderable material-instance recolor/highlight claims are not supported by this source.
