# Source Card: Arm GPU Best Practices Developer Guide - Fragment shading: Blending

## Summary
This first-party Arm developer guide (document 101897, v3.4) explains how alpha blending interacts with Arm Mali/Immortalis GPU overdraw-removal hardware. It establishes that enabling blending disables key hidden-surface-removal optimizations (early ZS testing and Forward Pixel Kill), so each overlapping transparent layer adds fragment overdraw that cannot be culled. It recommends disabling blending for opaque objects, minimizing/monitoring blended layer counts per pixel, splitting UI elements into opaque vs transparent parts, and preferring unorm over floating-point blend formats.

## Key evidence
- (ev_001) Blending disables important overdraw-removal optimizations, namely early ZS testing and FPK, hurting UI and multi-layer 2D sprite workloads most.
- (ev_002) High per-pixel blended-layer counts burn GPU cycles even with simple shaders because every fragment must be processed; monitor them.
- (ev_003) Splitting UI elements into opaque and transparent portions lets early ZS or FPK remove overdraw beneath the opaque parts.
- (ev_004) Blending and alpha-to-coverage should always be disabled for opaque objects.
- (ev_005) Blending is on-chip-efficient on Arm GPUs (dstColor in tile buffer) but format-sensitive; prefer unorm and note only simple FP16/R11G11B10 blends are accelerated.

## Limitations
- This specific page documents blending mechanics; it does not itself spell out the explicit opaque-front-to-back / transparent-back-to-front draw order or the discard/alpha-test hidden-surface-removal statements (those appear on adjacent pages of the same guide).
