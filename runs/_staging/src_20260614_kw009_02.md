# Source Card: The Mali GPU: An Abstract Machine, Part 2 — Tile-based Rendering

## Summary
This Arm Community engineering blog post by Peter Harris explains how Mali mobile GPUs use tile-based rendering: geometry is binned into 16x16 pixel tiles in a geometry pass, then each tile is shaded to completion in a fragment pass with its full color/depth/stencil working set held in fast on-chip RAM. The article establishes the bandwidth-energy motivation for this design (external DRAM ~120mW/GByte/s versus on-chip memory roughly an order of magnitude cheaper) and the resulting optimizations: cheap blending, Transaction Elimination via CRC compare, and discardable depth/stencil. It is a foundational, vendor-authoritative architecture description for reasoning about mobile-GPU-performant rendering patterns.

## Key evidence
- (ev_001) Mali bins the screen into 16x16 pixel tiles and builds per-tile primitive lists in a geometry pass.
- (ev_002) Each shader core shades one 16x16 tile to completion before the next, in the fragment pass.
- (ev_003) Framebuffer working set is ~16MB at 1080p and ~64MB at 4K2K (32bpp color + 32bpp depth/stencil).
- (ev_004) External DRAM costs ~120mW per 1GByte/s; on-chip tile memory is ~an order of magnitude cheaper.
- (ev_005) A whole tile's working set fits in fast on-chip RAM, making blending fast and power-efficient.
- (ev_006) Transaction Elimination skips a tile's color write via CRC compare when contents are unchanged.
- (ev_007) Depth/stencil tile contents are never written back if discarded (glDiscardFramebufferEXT / glInvalidateFramebuffer).

## Limitations
- Published 2014 and Mali-specific; energy and bandwidth figures are order-of-magnitude estimates that vary by system design, and the post predates newer Mali/Immortalis generations.
