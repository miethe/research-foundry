# Source Card: Automating Crochet Patterns for Surfaces of Revolution

## Summary
This paper (Martinez & Taylor Lipnicki, 2023) presents a deterministic algorithm and published program that automatically generates polished, human-readable crochet patterns for surfaces of revolution. Given a positive function f(x) rotated about the x-axis plus stitch/row gauge and a scale, it uses the arclength integral to place rows and a circumference-times-gauge formula to set stitch counts, then distributes increases/decreases via a "remainder method" optimized by a pairwise distance measure. It is the clearest reproducible deterministic-math-to-crochet pipeline in the source set, with runnable Sage/CoCalc code.

## Key evidence
- (ev_001) Inputs are f(x) (positive on (a,b), f'(x) defined on [a,b]) rotated about the x-axis, x-bounds, stitch gauge S, row gauge R, and scale; scope is axially symmetric shapes only.
- (ev_002) Row landmarks placed by dividing total arclength L = (scale*R/4) * integral sqrt(1+(f'(x))^2) dx, measured in rows (~16.162 in the example).
- (ev_003) Each landmark x_i solved by "guess and check" to two-decimal (0.01) precision, arclength always measured from x0 = a.
- (ev_004) Stitch count is deterministic: s(i) = round(2*pi * scale * (S/4) * f(x_i)) - circumference times stitch gauge.
- (ev_005) Program flags an error when row-to-row stitch count more than doubles/halves; recommends adding a positive constant to f(x) or adjusting scale.
- (ev_006) Output is standard crochet notation with per-row counts; worked example over [-3,1] gives 17 rows (0-16), stitches 6 to 51.
- (ev_007) Code is published and runnable: written in Sage, free on CoCalc, source at github.com/meganmartinez/math_crochet.

## Limitations
- Scope is restricted to surfaces of revolution (circular cross-sections); no general mesh or segmentation. The paper text does not state an explicit software license (the AGPL-3.0 claim from the original key points could not be confirmed from this source). Numeric landmark solving is "guess and check" to 0.01, so features closer than 0.01 are not resolved.
