# Source Card: inotify(7) — Linux manual page

## Summary
The canonical Linux man-pages reference for the inotify filesystem-monitoring API. It establishes the core operational semantics and caveats that any file-watch-based hook or watcher must contend with: event coalescing (which breaks reliable counting), bounded queues with a guaranteed overflow signal, per-user resource limits, non-recursive directory watching, and inherently racy rename pairing. These are primary, authoritative constraints for designing robust file-watch hooks.

## Key evidence
- (ev_001) Identical successive unread events are coalesced, so inotify cannot reliably count file events.
- (ev_002) Events over the queue limit are dropped, but an IN_Q_OVERFLOW event is always generated.
- (ev_003) The overflow event is delivered as IN_Q_OVERFLOW with wd = -1.
- (ev_004) Three per-real-user-ID tunables: max_user_instances, max_user_watches, max_queued_events.
- (ev_005) Directory monitoring is not recursive; subdirectories need their own watches.
- (ev_006) IN_MOVED_FROM/IN_MOVED_TO rename matching is inherently racy (non-consecutive, non-atomic, possibly absent).
- (ev_007) Pre-Linux-2.6.25 coalescing compared most-recent vs oldest-unread event (historical bug).

## Limitations
- Describes inotify-specific behavior on Linux only; macOS (FSEvents/kqueue) and Windows (ReadDirectoryChangesW) have different semantics. Reflects Linux man-pages 6.18 (2026-02-14); some numeric defaults (e.g., max_queued_events default) are kernel/version dependent and not captured verbatim here.
