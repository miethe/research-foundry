# Source Card: fanotify(7) — Linux manual page

## Summary
The fanotify(7) Linux man page documents the kernel's filesystem event notification API. It establishes that directory marks are non-recursive and that the FAN_CREATE-then-mark pattern for tracking new subdirectories is inherently racy, while mount- and filesystem-scoped marks (FAN_MARK_MOUNT / FAN_MARK_FILESYSTEM) provide race-free whole-tree monitoring. It also documents FID-reporting modes (FAN_REPORT_FID / FAN_REPORT_DIR_FID) that identify objects by file handle rather than file descriptor, and a documented security caveat tied to CAP_SYS_ADMIN.

## Key evidence
- (ev_001) Fanotify directory monitoring is not recursive; subdirectories require additional marks.
- (ev_002) FAN_CREATE-then-add-mark for new subdirectories is racy and can lose events occurring before the mark is set.
- (ev_003) FAN_MARK_MOUNT and FAN_MARK_FILESYSTEM enable race-free whole-directory-tree monitoring.
- (ev_004) FAN_REPORT_FID / FAN_REPORT_DIR_FID identify filesystem objects by file handle instead of file descriptor.
- (ev_005) Event-generated file descriptors bypass read/write authorization checks, a security risk under CAP_SYS_ADMIN.

## Limitations
- Reflects Linux man-pages 6.18 (dated 2026-02-14); fanotify behavior is kernel-version dependent and some features (e.g., FAN_REPORT_FID, FAN_MARK_FILESYSTEM) were introduced in specific kernel versions noted elsewhere in the page (not all captured here).
