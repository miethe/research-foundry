# Source Card: githooks Documentation — Git Reference Manual

## Summary
The Git reference manual's githooks page is the canonical specification for Git's hook system. It documents that the post-* hooks (post-commit, post-merge, post-receive, post-checkout, post-rewrite) run synchronously after the real work is already complete, making them notification/trigger points rather than veto gates. Each hook differs in its parameters, stdin format, and execution preconditions, and only post-checkout can influence its command's exit status.

## Key evidence
- (ev_001) post-commit takes no parameters, is for notification, and cannot affect git commit's outcome.
- (ev_002) post-merge takes a single squash status-flag, cannot affect the merge, and is skipped on conflict failure.
- (ev_003) post-receive runs once on the remote after all ref updates, only if at least one ref updated, with one stdin line per updated ref.
- (ev_004) post-receive does not affect git receive-pack because it runs after the real work is done.
- (ev_005) post-checkout receives three params: old HEAD ref, new HEAD ref, and a branch(1)/file(0) flag.
- (ev_006) post-checkout uniquely has its exit status become the command's exit status.
- (ev_007) post-checkout also runs after git clone (unless --no-checkout) and git worktree add, with null-ref first param and flag always 1.

## Limitations
- Page reflects Git 2.54.0 (last updated 2026-04-20); exact parameter/behavior details can evolve across Git versions, so version pinning matters for precise claims.
