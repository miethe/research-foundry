# Source Card: Using the File System Events API (FSEvents Programming Guide)

## Summary
Apple's first-party FSEvents Programming Guide establishes that the File System Events API notifies consumers at directory granularity ("something in this directory changed") rather than per-file, so an application must scan the named directory to discover the actual changes. Normally only the exact directory in the event path is scanned, but several flags change this: kFSEventStreamEventFlagMustScanSubDirs (set when directory/subdirectory events coalesce) requires a recursive rescan, and dropped-event flags (KernelDropped / UserDropped) require a full scan because the kernel cannot say what changed. This source is authoritative for designing file-watch/rescan logic on macOS/Darwin.

## Key evidence
- (ev_001) FSEvents reports at directory granularity; the app must scan the directory to learn which files changed.
- (ev_002) Normally scan only the exact directory in the event path, except in three situations.
- (ev_003) Coalesced directory/subdirectory events set MustScanSubDirs, requiring a recursive rescan.
- (ev_004) Dropped-event flags (KernelDropped/UserDropped) require a full scan because what changed is unknown.
- (ev_005) Dropped events also set MustScanSubDirs, so checking the dropped flags is not needed to decide on a rescan.

## Limitations
- Archived document (last updated 2012-12-13); describes the classic FSEvents API and may not reflect newer macOS file-monitoring APIs.
- The commonly cited "not notified more often than once per latency window" coalescing wording is NOT stated on this page; here the latency parameter appears only in the code example, so that specific claim should be sourced elsewhere.
