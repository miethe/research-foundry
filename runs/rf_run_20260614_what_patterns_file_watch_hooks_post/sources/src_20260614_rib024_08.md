# Source Card: Using server-sent events — Web APIs | MDN

## Summary
The MDN "Using server-sent events" reference documents how the Server-Sent Events (SSE) Web API delivers a one-way, server-to-client event stream over HTTP. It establishes the protocol's core constraints: the required `text/event-stream` MIME type, automatic reconnection with `.close()` termination, the `retry` and `id` event-stream fields, and a notable connection-count limitation (6 per browser per domain) when not using HTTP/2.

## Key evidence
- (ev_001) SSE is a one-way server-to-client channel; a client cannot send events to the server over it.
- (ev_002) Connections auto-reconnect by default if closed, and are terminated explicitly via `.close()`.
- (ev_003) The server-side script must respond with the `text/event-stream` MIME type.
- (ev_004) The `retry` field sets reconnection time in milliseconds (integer-only); the `id` field sets the EventSource last event ID.
- (ev_005) Without HTTP/2, SSE is limited to 6 open connections per browser per domain; HTTP/2 negotiates simultaneous streams (default 100).
- (ev_006) Page last modified May 15, 2025.

## Limitations
- Community-maintained reference (MDN) rather than a formal standard; authoritative for web platform behavior but should be cross-checked against the WHATWG HTML/EventSource specification for normative wording. The 6-connection limit is a browser implementation constraint (marked "Won't fix"), not a protocol mandate.
