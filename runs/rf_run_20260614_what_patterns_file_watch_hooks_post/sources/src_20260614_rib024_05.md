# Source Card: watchexec(1) — manual page

## Summary
The official watchexec(1) man page (v2.5.1, by Félix Saparelli and Matt Green) documents the file-watching tool's runtime behavior, including its debounce model, polling fallback, run-delay, and process stop semantics. It establishes concrete defaults (50 ms debounce, 30 s poll interval, 10 s stop timeout) and explicitly warns that filesystem rename events cannot be tracked reliably or portably.

## Key evidence
- (ev_001) Default debounce is 50 ms; Watchexec waits up to that long after an event before acting, and 0 is discouraged.
- (ev_002) Rename handling is unreliable: old-vs-new is not portably knowable, 'half'/'unknown' renames occur, and renames can split across debouncing boundaries.
- (ev_003) --poll forces filesystem polling instead of native OS watching, defaulting to a 30 s interval.
- (ev_004) --delay-run sleeps for a specified time before running the command after an event is detected.
- (ev_005) --stop-timeout defaults to 10 s; 0 immediately force-kills the command.

## Limitations
- Behavior and defaults are version-specific (captured from v2.5.1); later releases may change defaults or option semantics. The rename-tracking caveat is inherent to OS-level filesystem watching, not a watchexec-specific bug.
