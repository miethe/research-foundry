# Source Card: Flows — CrewAI Documentation

## Summary
The CrewAI Flows documentation defines Flows as a structured, event-driven workflow feature that orchestrates multi-step AI processes built on CrewAI's crews and capabilities. It specifies the decorator-based control model — @start() entry points, @listen() event listeners, and @router() conditional dispatch — and two state-management styles: unstructured (dynamic self.state) and structured (Pydantic BaseModel, type-safe and validated).

## Key evidence
- (ev_001) CrewAI Flows is an event-driven feature for building structured, multi-step AI workflows.
- (ev_002) @start() marks entry points; all satisfied starts execute (often in parallel) at begin/resume.
- (ev_003) Starts can be unconditional, gated on a prior method/router label, or callable-conditioned.
- (ev_004) @listen() runs a method automatically when a specified upstream task emits output.
- (ev_005) @router() defines conditional routing on a method's output to dynamically control execution.
- (ev_006) Unstructured state stores everything on the Flow's state attribute with no strict schema.
- (ev_007) Structured state uses a Pydantic BaseModel for type-safe, validated, auto-completable state.
