# Source Card: Router - Load Balancing | liteLLM

## Summary
This is the official LiteLLM documentation for its Router (load-balancing) component. It establishes the menu of routing strategies (weighted simple-shuffle as the default, plus latency-based, rate-limit-aware/usage-based-v2, least-busy, and lowest-cost routing) and the reliability layer that wraps deployment selection: deployment ordering/priority, cooldowns for failing deployments, retries with exponential backoff for rate-limit errors, and cross-group fallbacks. It also documents a plugin point for custom routing strategies via `CustomRoutingStrategyBase`.

## Key evidence
- (ev_001) Default/production-recommended strategy is the weighted simple-shuffle pick, weighted by rpm/tpm.
- (ev_002) usage-based-routing-v2 routes to the lowest-TPM deployment, using Redis to track usage in production.
- (ev_003) Latency-based routing picks lowest response time; least-busy picks fewest ongoing calls.
- (ev_004) Cost-based routing picks the lowest-cost healthy deployment under rpm/tpm limits, via the cost map.
- (ev_005) Cooldowns remove a deployment after a configurable number of per-minute failures.
- (ev_006) Retries use exponential backoff for RateLimitError and immediate retry for generic errors.
- (ev_007) Deployment `order`/priority is exhausted in sequence before falling through to configured fallbacks.
