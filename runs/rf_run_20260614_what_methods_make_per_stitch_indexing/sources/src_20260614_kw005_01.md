# Source Card: Language and Tool Support for 3D Crochet Patterns: Virtual Crochet with a Graph Structure (HPI Technical Report 137)

## Summary
This 2021 Hasso-Plattner-Institut technical report (Seitz, Lincke, Rein, Hirschfeld) defines a domain-specific graph representation for crochet patterns in which each stitch insertion point is a node and inter-stitch connections are typed edges ('previous' for order, 'insert' for the worked-into relationship). It establishes a uniform, topology-based scheme for increases/decreases (edge multiplicity), unifies rows and rounds into per-node 'layers,' enforces crochetability through valid-connection-only editing, and demonstrates a force-directed 3D layout prototype that positions stitch nodes as the pattern graph is built.

## Key evidence
- (ev_001) Patterns are modeled as a graph: insertion points are nodes, inter-stitch connections are edges, and traversal follows the yarn path.
- (ev_002) Edges are typed — a 'previous' edge encodes order; an 'insert' edge encodes the worked-into (insertion) relationship and its insertion type.
- (ev_003) Increases/decreases are a uniform edge-multiplicity rule (multiple out-going vs. multiple incoming insert edges) rather than positional counting.
- (ev_004) Rows and rounds are unified into 'layers,' stored as a layer-number property per node, with no structural change required.
- (ev_005) Crochetability is enforced structurally: connections can only target already-existing nodes, and the editor permits only valid connections.
- (ev_006) A force-directed 3D layout (3D Force-Directed Graph / Three.js + WebGL + D3 physics, Dwyer's algorithm) positions stitch nodes automatically.
- (ev_007) Nodes carry layer + 'start' (first-of-layer) + previous/next/inserts properties, yielding per-stitch layer-plus-position addressing.

## Limitations
- Describes a research proof-of-concept projectional editor and prototype, not a production tool; the 3D layout is a generic force-directed approximation rather than a yarn-physics-accurate model. Validity enforcement in the prototype is via UI restriction; the authors note a constraint solver would be needed to verify arbitrary externally-built graphs.
