# Source Card: Representing Crochet with Stitch Meshes

## Summary
This SCF '20 paper (Guo, Lin, Narayanan, McCann; CMU) extends the knitting "stitch mesh" tile paradigm to crochet by introducing a special edge type for the on-hook "current loop" and a four-label directed-edge scheme (Next/Previous for short-term consecutive dependencies, Future/Past for long-term loop-through-fabric dependencies). It defines an explicit graph/mesh intermediate representation that, unlike international crochet notation, captures how yarn is used within each stitch — enough to generate cube, sphere, and Stanford bunny patterns from triangle meshes. The modeled tile set is deliberately a subset (chain, slip stitch, single crochet, sc2tog, sc3tog plus utility faces).

## Key evidence
- (ev_001) A special edge type captures the on-hook "current loop," adapting stitch meshes to crochet.
- (ev_002) Every modeled crochet stitch begins and ends with a single leading loop, yielding a well-defined yarn-path order.
- (ev_003) International crochet notation omits explicit yarn-use-within-stitch info, making it inconvenient for computer authoring.
- (ev_004) Four directed edge labels (Next/Previous, Future/Past) encode short- and long-term stitch dependencies.
- (ev_005) Short-term edges are orange (crossed by leading loop + free yarn); long-term edges are purple (crossed only by loops).
- (ev_006) Increases split one future edge into two via an 'inc' utility face; decreases use sc2tog and sc3tog.
- (ev_007) Only chain, slip stitch, single crochet, sc2tog, and sc3tog were modeled — the tile set is explicitly incomplete.
- (ev_008) Cube, sphere, and Stanford bunny patterns were generated, the latter two from triangle meshes via the Narayanan et al. 2019 pipeline.

## Limitations
- The modeled stitch tile set is explicitly a subset of crochet (5 base stitches plus utility faces); stitches such as front-post-double-crochet cannot be represented by the framework, and yarn-relaxation/simulation is described by the authors as ad-hoc.
