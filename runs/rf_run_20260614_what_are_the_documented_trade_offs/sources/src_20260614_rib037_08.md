# Source Card: The Story of Scalar

## Summary
A first-party GitHub engineering retrospective (by Git maintainers Derrick Stolee and Victoria Dye) on how Scalar evolved from a filesystem-virtualization tool (VFS for Git) into a thin command-line shell around core Git features. It documents concrete scaling trade-offs: Git does not scale to large monorepos out of the box, and the bundle of features needed (partial clone, sparse-checkout, background maintenance, advanced config) carries its own performance pitfalls — most notably a quadratic-time sparse-checkout pattern matcher that cone mode replaced. It also explains the external forcing function (Apple deprecating macOS kernel virtualization features) that pushed the architecture toward core Git.

## Key evidence
- (ev_001) Pre-optimization sparse-checkout pattern matching was quadratic-time and took 40 minutes on one large repo.
- (ev_002) Cone-mode sparse-checkout brought git status down to three or four seconds, near VFS for Git's typical case.
- (ev_003) Scaling Git relies on a bundle of features (partial clone, sparse-checkout, background maintenance, advanced config) that scalar clone configures, implying plain Git does not enable them by default.
- (ev_004) Apple's deprecation of macOS kernel filesystem-virtualization features forced abandoning the VFS-for-Git approach.
- (ev_005) After the pivot, Scalar became only a command-line interface on top of Git rather than a virtualization layer.

## Limitations
- Vendor-authored narrative (GitHub/Git maintainers describing their own product); the 40-minute and "three or four seconds" figures are illustrative anecdotes from specific large repositories, not a controlled, reproducible benchmark.
