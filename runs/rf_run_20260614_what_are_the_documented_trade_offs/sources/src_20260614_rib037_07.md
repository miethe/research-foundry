# Source Card: PostgreSQL Documentation: Introduction to Multiversion Concurrency Control (MVCC)

## Summary
The official PostgreSQL documentation (Chapter 13.1) establishes that PostgreSQL maintains data consistency through Multiversion Concurrency Control (MVCC), where each SQL statement sees an immutable snapshot of the data as it existed at an earlier point. This per-session snapshot model provides transaction isolation and ensures that reads never block writes and writes never block reads. The docs make a qualitative (non-benchmarked) claim that proper MVCC use generally outperforms locking, and note the guarantee holds even under Serializable Snapshot Isolation (SSI).

## Key evidence
- (ev_001) Data consistency is maintained via a multiversion model where each SQL statement sees a snapshot/database version of data as of an earlier time.
- (ev_002) The snapshot model prevents statements from viewing inconsistent data from concurrent updates, providing transaction isolation per session.
- (ev_003) Read locks do not conflict with write locks: reading never blocks writing and writing never blocks reading.
- (ev_004) MVCC eschews traditional locking methodologies to minimize lock contention for reasonable multiuser performance.
- (ev_005) Proper MVCC use will generally provide better performance than locks (qualitative claim, no benchmarks).
- (ev_006) The MVCC guarantee is preserved even at the strictest isolation level via Serializable Snapshot Isolation (SSI).

## Limitations
- The performance comparison ("better performance than locks") is qualitative and not accompanied by benchmark data; it should be cited as a documented design assertion, not a measured result.
