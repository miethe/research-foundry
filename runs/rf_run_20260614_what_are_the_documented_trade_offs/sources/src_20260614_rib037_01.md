# Source Card: Git - hash-function-transition Documentation (SHA-1 to SHA-256)

## Summary
This is the official Git project design document describing Git's transition from SHA-1 to SHA-256 for content addressing. It establishes that Git is fundamentally a content-addressable filesystem that names objects by hash, that blob objects have identical SHA-1 and SHA-256 content because they reference no other object, and that the move to SHA-256 is motivated by the 2017 SHAttered SHA-1 collision while preserving Git's content-addressable model and supporting incremental, interoperable adoption.

## Key evidence
- (ev_001) Git is a content-addressable filesystem that uses SHA-1 to name files, directories, and revisions.
- (ev_002) A blob refers to no other object, so its SHA-1 content and SHA-256 content are the same value.
- (ev_003) The 2017 SHAttered attack proved a practical SHA-1 collision; SHA-1 remains weak, motivating SHA-256.
- (ev_004) An object's SHA-256 name is the SHA-256 of its type, length, nul byte, and SHA-256 content, preserving content addressing.
- (ev_005) The transition is incremental (one repo at a time) and interoperable (SHA-256 repos can push/fetch with SHA-1 servers).

## Limitations
- This is a design/specification document; some stated transition goals describe the intended end state and may not all be fully shipped in stable Git as of access. Two key_points supplied to the carder (explicit "immutability"/"acyclicity guarantee" of the commit graph, and quick diffing by comparing hashes/ignoring identical object names) could NOT be confirmed verbatim in this document and were therefore omitted to avoid fabrication.
