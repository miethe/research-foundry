# Source Card: Appropriate Uses For SQLite

## Summary
The official SQLite documentation page "Appropriate Uses For SQLite" defines where the embedded, file-first database is and is not a good fit. It establishes the core concurrency model — unlimited simultaneous readers but exactly one writer at any instant — and gives concrete decision guidance: many concurrent writers that cannot take turns point to a client/server engine, while low-to-medium write concurrency works because write transactions take milliseconds. It also quantifies scale limits (suitable below ~100K website hits/day; max DB size 281 TB; consider client/server when data approaches the terabyte range).

## Key evidence
- (ev_001) SQLite allows unlimited simultaneous readers but only one writer at any instant — the file-first concurrent-write breakpoint.
- (ev_002) When many threads/processes must write at the same instant and cannot queue/take turns, the docs direct readers to a client/server database engine.
- (ev_003) A write transaction typically takes milliseconds, so multiple writers can take turns under low-to-medium write concurrency.
- (ev_004) Websites with fewer than 100K hits/day should work fine with SQLite.
- (ev_005) sqlite.org itself serves ~400K-500K HTTP requests/day, ~15-20% touching the database.
- (ev_006) Max SQLite database size is 281 terabytes (2^48 bytes); consider a centralized client/server DB when data may reach the terabyte range.

## Limitations
- Vendor/project documentation: authoritative on SQLite's own design and limits, but reflects the project's framing of trade-offs rather than independent benchmarking. Figures (hits/day, request volumes) are illustrative guidance, not hard thresholds. Last updated 2025-05-31; verify against the current page for later revisions.
