# Source Card: PostgreSQL 18 Documentation — Chapter 29: Logical Replication

## Summary
The official PostgreSQL 18 documentation defines logical replication as the replication of data objects and their changes based on a replication identity (usually a primary key), contrasting it with physical block-level replication. It operates on a publish/subscribe model where subscribers pull from publications and may re-publish for cascading topologies, and it begins with an initial snapshot copy followed by continual streaming of subsequent changes in publisher commit order. This source establishes the canonical mechanism and documented use cases (incremental change delivery, per-change trigger firing) underpinning CDC and audit workflows.

## Key evidence
- (ev_001) Logical replication replicates data objects and their changes based on a replication identity (usually a primary key), in contrast to physical block-level replication.
- (ev_002) It uses a publish/subscribe model where subscribers pull from publications and may re-publish for cascading replication.
- (ev_003) On start it snapshots the publisher table, copies it to the subscriber, then continually streams subsequent changes.
- (ev_004) The subscriber applies changes in publisher order, guaranteeing transactional consistency within a single subscription.
- (ev_005) Documented use cases include sending incremental changes as they occur and firing triggers per individual change on arrival.
