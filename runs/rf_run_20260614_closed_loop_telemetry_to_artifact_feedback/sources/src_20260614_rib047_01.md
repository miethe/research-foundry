# Source Card: How to run human-in-the-loop evals for LLM apps

## Summary
A Braintrust vendor article describing how to operationalize human-in-the-loop evaluation for LLM applications inside a single tracing/scoring/CI workflow. Its core thesis is a closed feedback loop: reviewers score at the span level, human-reviewed failures convert into permanent eval dataset rows (regression tests), human scores calibrate LLM-as-a-judge scorers, and the same scorers run across offline evals, CI/CD, and live traffic as release quality gates. Because it is published by the vendor, capability claims are self-reported and feature-tied.

## Key evidence
- (ev_001) Reviewers assign scores to individual spans, not just final output, to localize where a failure began.
- (ev_002) Human-reviewed failures become permanent eval dataset rows, turning each production failure into a regression test.
- (ev_003) Human scores act as ground truth to calibrate LLM-as-a-judge scorers via same-trace comparison.
- (ev_004) The same scorers run across offline evals, CI/CD, and live traffic so human feedback informs release decisions.
- (ev_005) A GitHub Action posts eval results to PRs and can block merges when quality thresholds are missed.
- (ev_006) Without this loop, review feedback stays trapped in spreadsheets/labeling tools and never reaches the eval pipeline, CI/CD, or monitoring.

## Limitations
- Vendor-authored marketing/docs content: the closed-loop and span-level capabilities are presented as product features of Braintrust, so claims are self-reported rather than independently benchmarked. The mechanisms (lagging signal to leading guardrail, ground-truth calibration, unified scorers as release gates) are described as design patterns and are generalizable, but specific quantitative outcomes are not provided.
