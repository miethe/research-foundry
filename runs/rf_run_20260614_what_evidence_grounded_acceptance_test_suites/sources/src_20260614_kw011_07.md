## Summary
This is the author's own ChatGPT-generated two-track research plan for the KnitWit "crochet OS" app, covering a Pattern->3D visualizer and an inverse 3D-mesh->crochet-pattern generator. It establishes explicit end-states and acceptance criteria for both engines, mandates a machine-validated Crochet IR with concrete validation rules, and specifies an inverse-engine evaluation protocol with a primitive mesh test set and complexity metrics. It also names the project's top risks (PDF parsing generality, gauge/tension variability) and the mitigation of validator-gated structured authoring.

## Key evidence
- (ev_001) Crochet IR validation rules: per-round stitch-count invariants, increase/decrease legality, and piece-closure constraints form the spine of acceptance.
- (ev_002) Visualizer acceptance criteria: geometric fidelity via Hausdorff distance / volumetric difference plus topological correctness (no broken loops, correct stitch counts per round).
- (ev_003) Inverse-engine acceptance criteria: crochetability (each round resolves, no ambiguous instructions) and shape-match within tolerance bands.
- (ev_004) Inverse generator must guarantee bounded shaping: no impossible stitch-count transitions and bounded inc/dec rate.
- (ev_005) Mesh test set of 20-50 simple primitives scored on shape distance and pattern-complexity metrics (max inc/dec rate per round, total rounds, stitch-count variance).
- (ev_006) Inverse preconditions stated explicitly: watertight mesh, user seed point, target stitch size and gauge assumptions.
- (ev_007) Flags the "PDF importer trap" and gauge/tension variability as top risks; recommends structured authoring over general PDF/NL parsing.
