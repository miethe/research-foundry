# Source Card: How to use model version aliases - Vertex AI Model Registry

## Summary
This first-party Google Cloud documentation page defines a model alias as a mutable, user-named, model-scoped reference to a specific model version, analogous to Docker tags or Git branch references. Aliases let callers fetch or deploy a version by reference without knowing its version ID. Exactly one version must always carry the reserved "default" alias (auto-assigned to the first version created), while users may create additional custom aliases to mark, for example, the stable version ready for deployment.

## Key evidence
- (ev_001) A model alias is a mutable, user-named reference to a model version, unique within a model resource; mutable because it can be moved between versions.
- (ev_002) Aliases let you fetch or deploy a version by reference without knowing its version ID, like Docker tags or Git branch references.
- (ev_003) The first version of a new model is automatically assigned the default alias, used when no version is specified.
- (ev_004) Exactly one version must carry the default alias at all times; otherwise it behaves like any user-defined alias.
- (ev_005) Version aliases must match `[a-z][a-z0-9-]{0,126}[a-z0-9]` (distinct from version_id), and exactly one default alias must exist per model.
- (ev_006) Aliases must be unique and assigned to only one version per model; reassigning an existing alias removes it from the prior version.
- (ev_007) The console alias marker shows at a glance the stable version ready for deployment; custom aliases can be created alongside the default.

## Limitations
- The source establishes custom aliases and the "stable/ready for deployment" framing, but the page text does not contain the specific suggested review-stage labels (e.g., "not started/in progress/under review/approved") or lifecycle labels (e.g., "experimental/staging/production") referenced in some summaries; those naming conventions are not asserted verbatim by this page and were not cited.
