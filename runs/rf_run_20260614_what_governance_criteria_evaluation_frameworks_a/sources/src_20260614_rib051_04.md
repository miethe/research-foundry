# Source Card: Model Registry Workflows | MLflow AI Platform

## Summary
The official MLflow documentation describes the Model Registry's model-lifecycle workflow. It documents the legacy stage-based model (Staging / Production / Archived / None) transitioned via `transition_model_version_stage()`, notes that Model Stages are deprecated as of MLflow 2.9.0 and will be removed in a future major release, and presents the replacement deployment tooling: model version aliases (named, multi-per-version references) and model version tags (status annotations such as `validation_status: pending`). The documented stage transitions are direct state updates with no built-in approval/reject or transition-request workflow.

## Key evidence
- (ev_001) MLflow's legacy lifecycle governs a model version via stages Staging, Production, or Archived (plus None).
- (ev_002) Stage transitions use `transition_model_version_stage`, accepting Staging|Archived|Production|None.
- (ev_003) Model Stages are deprecated as of MLflow 2.9.0 and will be removed in a future major release.
- (ev_004) Replacement aliases are named references and, unlike stages, allow multiple aliases per version for A/B testing and rollout.
- (ev_005) Replacement tags annotate version status, e.g. a `validation_status` tag moving from `pending` to `passed`.

## Limitations
- This is the "latest" docs URL, so stage/alias semantics reflect a current (2025) MLflow release; exact version-specific behavior may drift. The page does not verbatim state the "one stage per version at a time" rule (implied, not quoted) and describes only direct stage updates with no documented approval/reject transition-request workflow.
