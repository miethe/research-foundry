# Source Card: Prompt Version Control - Langfuse Docs

## Summary
This official Langfuse documentation page describes how prompt version control and deployment work in the Langfuse platform. Each prompt update creates a version with an auto-assigned version ID, while mutable labels (e.g., `production`, `staging`, `latest`) act as pointers that deploy specific versions to environments, tenants, or experiments. Fetching without a label serves the `production`-labeled version, rollbacks are performed by reassigning `production` to a prior version, and protected labels add RBAC-based governance over which roles may change deployment-controlling labels.

## Key evidence
- (ev_001) Each prompt version is automatically assigned a version ID; labels can additionally be assigned for a custom versioning scheme.
- (ev_002) Labels map prompt versions to environments, tenants, or experiments.
- (ev_003) Deploying a prompt version means assigning it the `production` (or another environment) label.
- (ev_004) Fetching without a label serves the version with the `production` label by default.
- (ev_005) The `latest` label always points to the most recently created version.
- (ev_006) Rollback = reassign the `production` label to a previous version in the UI; that version is then served by default in the SDKs.
- (ev_007) Protected prompt labels let admins/owners prevent labels like `production` from being modified or deleted, governing deployment via RBAC.
- (ev_008) Labels are assignable via UI and via the Python and JS/TS SDKs (on create and on update of existing versions).
