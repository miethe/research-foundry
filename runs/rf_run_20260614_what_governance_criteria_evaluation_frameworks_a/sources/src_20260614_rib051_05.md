# Source Card: Manage model lifecycle using the Workspace Model Registry (legacy)

## Summary
This official Databricks documentation page describes the lifecycle governance model of the legacy Workspace Model Registry: four model-version stages (None, Staging, Production, Archived), a request/approve workflow that lets unprivileged users request transitions while reviewers Approve/Reject/Cancel them, and an Activities log that captures the full transition lineage for auditing. The page is explicitly labeled legacy and directs Unity-Catalog-enabled workspaces to use Models in Unity Catalog instead.

## Key evidence
- (ev_001) Four stages (None, Staging, Production, Archived); Staging = testing/validation, Production = reviewed-and-deployed live scoring, Archived = inactive.
- (ev_002) A user without stage-transition permission can request a transition, which appears in Pending Requests.
- (ev_003) Reviewers click Approve, Reject, or Cancel to act on a pending request; the request creator can also cancel.
- (ev_004) The Activities section records requested/approved/pending/applied transitions as a lifecycle lineage for auditing.
- (ev_005) The registry is marked legacy; Databricks steers Unity-Catalog-enabled workspaces to Models in Unity Catalog.

## Limitations
- This describes a feature Databricks explicitly labels "legacy" and deprecates in favor of the Unity Catalog model registry / model aliases; the stage-based governance model may not reflect Databricks' current recommended approach. Documentation last updated February 2026.
